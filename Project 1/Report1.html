Summary

The project description covered most of the information we needed in order to design the ChocAn software. We made some assumptions on the critical actors for each use case, intending to be as thorough as possible so that all necessary parties were included. The glossary was produced as the project description was read and updated as our understanding of the project grew. The use cases were roughly made while reading as well, and then they were completed. Then after all the use cases were created and fully described, the project description was reread so as to assure that all necessary features were included. Each use case is featured in the html, separated by black lines. The Main Success Scenarios were implemented using a combination of ordered and unordered lists. Bitbucket was used to share the workload and maintain an up to date and collaborative codebase. This was essential for sharing ideas for the report.html file and allowing each team member to contribute.

Glossary

ChocAn: an organization that helps anyone addicted to chocolate
Operator: any person who edits data for ChocAn who is not a Manager, Provider, or Member
Manager: a person who is directly a part of the leadership of ChocAn
ChocAn Data Center: the data base in charge of verifying the identity of Members, Providers, Managers, and stores all service & billing data 
Members: people addicted to chocolate who are seeking consultations and treatments
Member Number: a nine-digit number associated with a Member’s account			 
Membership Card: a plastic card that has the Member’s name & Member number
Provider: any healthcare professional contracted by ChocAn
Provider Number: an individual ID number given to each Provider
Provider Directory: a list of all possible services tand their corresponding service number and service fee 
Terminal: a monitor that has access to the ChocAn software
ChocAn Software: the overall program that connects terminals & the ChocAn Data Center
Acme: Acme Accounting Services (third-party organization) responsible for financial procedures
Provider Banking: non-human 3rd party actor that receives billing information to be sent to the Provider’s banking software

Use Case Diagram

Use Case Description

Use Case 1: Verification of Member Identity

Context: A Member's Identity must be verified when seeking services

Actors: Providers, ChocAn Data Center

Main Success Scenario:

  1. The Provider must log into the Terminal by inputting their Provider Number
  2. The Member hands over their membership card to the Provider
  3. The Provider swipes the membership card on the Terminal
  4. The ChocAn Software dials the ChocAn Data Center to validate the Member’s Identity
    1. The word “Validated” appears on the Terminal screen
		
Extensions:
	4.a. The ChocAn Software fails to validate the Member's Identity
		4.a.1: "Invalid Number" appears on the Terminal screen
		4.a.2: If the Member hasn't paid membership fees for at least the past month, the Member's status is "suspended"
			4.a.2.i: "Member Suspended" appears on the Terminal Screen

Use Case 2: Billing ChocAn

Context: Providers must input information to bill ChocAn after services are finished

Actors: Provider, ChocAn Software

Main Success Scenario:

  1. Provider must log into the Terminal by inputting their Provider Number
  2. Member gives their Member Card to the Provider
  3. The Provider swipes the Membership Card on the Terminal
    1. Provider may manually key in the Member Number
  4. The Member Number is verified as stated in Use Case 1
  5. The Provider keys in date of service in the format MM-DD-YYYY
  6. The Provider must look up the six-digit service code in the Provider Directory for the service provided
  7. The Provider must key in the service code
  8. The ChocAn Software checks the code:
    1. If valid, the name of the service corresponding to the code appears on the Terminal screen (up to 20 characters)
    2. The ChocAn Software asks the Provider to verify the service keyed in
  9. The Provider may enter comments about the service
  10. The ChocAn Software writes the information to a report:
    1. Current date and time (MM–DD–YYYY HH:MM:SS)
    2. Date service was provided (MM–DD–YYYY)
    3. Provider number (9 digits)
    4. Member number (9 digits)
    5. Service code (6 digits)
    6. Comments (100 characters) (optional)
  11. The ChocAn Software searches the Provider Directory to determine the fee for the service provided
  12. The fee for the service provided is displayed on the Terminal screen
		
Extensions:
	8a. If the software cannot identify the code, the code is nonexistent
		8a.1: An “Error” message is printed on the terminal screen

Use Case 3: Billing Verification by Provider

Context: Providers must complete a form to verify billing information.

Actors: Provider, ChocAn Software

Main Success Scenario:

  1. The Provider must enter the following information:
    1. The current date and time
    2. The date the service was provide
    3. Member name and number
    4. The service code for the service provided
    5. The fee for the service provided
  2. After 7 days, the Provider must total all fees to verify the amount they are to be paid by ChocAn

Use Case 4: Main Accounting Procedure

Context: The ChocAn Software runs a weekly report by midnight on Friday summarizing services.

Actors: ChocAn Software, ChocAn Data Center

Main Success Scenario:

  1. The ChocAn Software reads the week’s file of services provided and compiles it into different reports including:
    • Weekly Summary Report
    • Member Service Reports
    • Provider Service Reports

Use Case 5: Provider Requests

Context: Providers can request a copy of the Provider Directory.

Actors: Provider, ChocAn Software

Main Success Scenario:

  1. The Provider must input their Provider Number into the Terminal
  2. The Provider makes a request from the Software for the complete Provider Directory
  3. The Software dials the ChocAn Data Center to access the Provider Directory
  4. The ChocAn Software sends an email to the Provider with the directory as an attachment

Use Case 6: Member Service Report

Context: Members who have consulted a ChocAn Provider during that week recive a list of services provided, sorted in order of service date

Actors: ChocAn Software

Main Success Scenario:

  1. The software compiles a summary report of services provided to each Member containing the following information
    • Member name (25 characters)
    • Member number (9 digits)
    • Member street address (25 characters)
    • Member city (14 characters)
    • Member state (2 letters)
    • Member ZIP code (5 digits)
    • For each service provided, the following details are required
      • Date of service (MM-DD-YYYY)
      • Provider name (25 characters)
      • Service name (20 characters)
  2. The software sorts the services by date of service
  3. The report is sent as an email to the Member

Use Case 7: Provider Service Report

Context: Each Provider billing ChocAn in a certain week is sent an email with the list of services provided that week. The information on the report is the same as entered on the Provider’s form.

Actors: Provider, ChocAn Software

Main Success Scenario:

  1. The ChocAn Software creates a report containing the following information for each Provider:
    • Provider name (25 characters)
    • Provider number (9 digits)
    • Provider street address (25 characters)
    • Provider city (14 characters)
    • Provider state (2 letters)
    • Provider ZIP code (5 digits)
    • For each service provided, the following details are required:
      • Date of service (MM–DD–YYYY).
      • Date and time data was received by the computer (MM–DD–YYYY HH:MM:SS).
      • Member name (25 characters).
      • Member number (9 digits).
      • Service Code (6 digits)
      • Fee to be paid (up to $999.99)
  2. The software sums the following information and appends it to the report:
    • Total number of consultations with Members (3 digits)
    • Total fee for the week (up to $99,999.99)
  3. The reports are then sent as an attatchment in an email to the Provider
  4. A record consisting of EFT data is written to a disk and stored

Use Case 8: Weekly Summary Report

Context:The Summary Report lists every Provider being paid that week, the number of consultations each, and the total fee for each Provider. Combined numbers are included such as the amount of Providers paid, amount of total consultations, and the combined fee

Actors: ChocAn Software, Manager

Main Success Scenario:

  1. The system compiles a list including:
    1. The list of all Providers being paid
    2. The number of consultations for each Provider
    3. The fee paid to each Provider
    4. The total number of Providers paid
    5. The total number of Consultations
    6. The combined fee paid to all Providers
  2. This report is sent as an attachment in an email to the Manager

Use Case 9: Interactive Mode

Context: The program needs to be able to keep track of current users, and update their information as needed.

Actors: Operators

Main Success Scenario:

  1. As needed Operators:
    1. Add Members:
    2. Delete Members who’ve resigned
    3. Update Member records
    4. Add new Providers
    5. Delete Providers
    6. Update Provider information

Use Case 10: Manager Requests

Context:Managers can run summary reports and access old reports at any time

Actors: Manager, ChocAn Software

Main Success Scenario:

  1. The Manager must use their terminal to request the ChocAn Software to run a specific report
  2. Report types include
    1. Member Service Report
    2. Provider Service Report
    3. Weekly Summary Report
  3. Reports will be compiled in the methods stated in their individual Use Cases’ and then emailed as an attachment to the Manager
